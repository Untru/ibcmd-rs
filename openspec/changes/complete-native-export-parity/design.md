# Дизайн: полная побайтовая совместимость экспорта УТ и БСП

## 1. Цель

Довести прямую выгрузку конфигурации из MSSQL через `ibcmd-rs` до полного
побайтового совпадения с нативным `ibcmd` на двух экспериментальных базах:

- `ut_ibcmd` — УТ 11.5.25.112, платформа 8.3.27;
- `bsp` — база БСП, используемая предыдущими сериями исправлений.

Итоговый критерий экспорта для обеих баз:

```text
different = 0
left_only = 0
right_only = 0
unchanged = total
```

Семантическое совпадение, нормализация XML и выборочные прогоны используются
только для диагностики. Они не могут заменить финальный побайтовый критерий.

После достижения точного экспорта отдельным этапом проверяется обратный маршрут:
XML -> одноразовый клон MSSQL -> нативный экспорт -> побайтовое сравнение.

## 2. Исходные данные на 23 июля 2026 года

### 2.1 Свежий снимок УТ

В `E:\ibcmd_lab\ut` уже существует Git-сравнение:

- рабочее дерево `ut_ibcmd` — выгрузка `ibcmd-rs` от 22 июля;
- всего в нативном дереве — `49 623` файла;
- совпадает — `45 527`;
- отличается — `3 597`;
- присутствует только в нативной выгрузке — `499`;

экспортёра, версиями инструментов и командами запуска.

### 2.2 Установленные первопричины

1. **499 пропущенных корневых XML метаданных.**
   `282` документов, `198` справочников, `9` планов видов характеристик,
   `7` бизнес-процессов и по одному объекту обработки, критерия отбора и задачи.
   Они дают около `87,5%` всех удалённых строк. Вероятная граница дефекта:
   `src/mssql_dump/metadata.rs` и маршрутизация в
   `src/mssql_dump/mod.rs`: нераспознанное тело метаданных возвращает `None`,
   ресурсы объекта выгружаются, а корневой XML молча пропускается.

2. **Формы.**
   Основные повторяющиеся потери: `ExcludedCommand`, `v8:Type`, `Type`,
   `Title`, `Column`, `DataPath`, `AllowedLength`, `StringQualifiers`.
   Наблюдаются замена имён событий на UUID, неверный namespace `TypeId`,
   потеря суффикса `.Balance` у `MainTable`, лишние пустые настройки и значения
   по умолчанию. Основные файлы:
   `src/mssql_dump/form_body.rs`, `src/mssql_dump/forms.rs`,
   `src/module_blob.rs`, `src/form_matrix.rs`.
   Обезличенные агрегаты полного корпуса и очередность первых исправлений
   приведены в
   [сводке по формам УТ](../../../docs/evidence/ut-form-signatures-20260723.md).

3. **MXL и другие шаблоны.**
   Главные кластеры: `document/format/f`, `formatIndex`, шрифты, границы,
   фоновые цвета и стили. Основные файлы:
   `src/mssql_dump/moxel.rs`, `src/module_blob.rs`.

Исправления, выполненные на БСП, закрывали отдельные варианты двоичного
представления и отдельные UUID. Они не доказывают универсальность на корпусе УТ.
Новые исправления запрещено привязывать к именам или UUID объектов конкретной
базы, кроме тестовых описаний входного корпуса.

### 2.3 Проблемы текущего процесса

- `scripts/01-export-ibcmd.bat` и `scripts/02-export-ibcmd-rs.bat` пишут в один
  каталог и превращают Git в неявный способ сравнения.
- `scripts/export-ibcmd-vs-ours.ps1` уже разделяет стороны, но не создаёт
  полный манифест запуска и не строит общую матрицу двух баз.
- Скрипт ожидает `dump-sources`, доступный только в сборке с
  `platform-oracle`, но проверяет лишь наличие любого `target\release`-файла.
- Старый полный `source-diff` от 3 июля показывает `56,9%`, а свежий Git-аудит
  — `91,7458%`; без протокола снимков эти числа легко смешать.
- `compatibility/matrix.json` описывает маршруты автономной конвертации, а не
  качество каждого файла реальной выгрузки.
- Нет машинно проверяемой таблицы `база × путь × семейство × статус × сигнатура`.

## 3. Архитектура проверочного контура

```text
одноразовый снимок БД
        |
        +--> native ibcmd ------> native/
        |
        +--> ibcmd-rs MSSQL ----> candidate/
                                  |
native/ + candidate/ --> raw source-diff
                                  |
                    +-------------+--------------+
                    |                            |
            XML signatures                per-path matrix
                    |                            |
                    +-------------+--------------+
                                  |
                         aggregate UT + BSP
                                  |
                    JSON + Markdown + backlog
```

### 3.1 Каталог запуска

Каждый запуск получает отдельный каталог:

```text
E:\ibcmd_lab\parity\<database>\<timestamp>_<git-sha>\
  manifest.json
  native\
  candidate_dump\
  candidate\
  native-report.json
  candidate-report.json
  raw-diff.json
  signatures.json
  matrix.json
  matrix.md
  stderr\
```

Ни одна сторона не имеет права писать в каталог другой стороны. Существующий
каталог запуска не перезаписывается.

### 3.2 Манифест

`manifest.json` хранит:

- версию схемы;
- имя базы и сервера без пароля;
- SHA исходного кода;
- dirty/clean состояние репозитория;
- платформу, путь и версию `ibcmd`;
- версию XML;
- точные аргументы команд без секретов;
- время начала и завершения каждого шага;
- коды завершения;
- сводки и хеши деревьев;
- признак полного или выборочного запуска;
- до- и послезапусковой отпечаток конфигурационного хранилища.

Пароли берутся только из переменных окружения и никогда не записываются.
Если отпечаток базы изменился между сторонами, результат помечается
недействительным.

### 3.3 Побайтовая матрица

Новая модель строится поверх существующего `SourceDiffReport`, без второго
обходчика файлов:

```text
ParityRun
ParityRow
ParityAggregate
ParityMatrix
```

Каждая строка содержит:

- базу и идентификатор запуска;
- относительный путь;
- семейство и вид исходника;
- подсказку объекта;
- `unchanged | different | left_only | right_only`;
- размеры и SHA-256 обеих сторон;
- XML-сигнатуры или ошибку разбора;
- несколько детерминированных примеров;
- владельца рабочего пакета и состояние исправления.

Агрегаты строятся по базе, семейству, виду файла, статусу и сигнатуре.
Сумма агрегатов обязана точно совпадать со сводкой `source-diff`.

### 3.4 Нормализованное сравнение

Для изменённых XML допускается дополнительная классификация:

- только форматирование;
- порядок узлов или атрибутов;
- значение;
- отсутствующий узел;
- лишний узел;
- namespace;
- ошибка разбора.

Поле `raw_status` остаётся неизменным. Нормализованное совпадение никогда
не переводит строку в побайтово успешную.

### 3.5 Две оси готовности

1. **SQL -> XML:** прямой экспорт `ibcmd-rs` против нативного `ibcmd`.
2. **XML -> SQL -> XML:** загрузка в одноразовый клон и повторный нативный
   экспорт.

Эти проценты и блокеры публикуются раздельно.

## 4. Стратегия исправлений

### 4.1 Сначала наблюдаемость

До массовых исправлений должны появиться:

- воспроизводимый полный запуск для УТ и БСП;
- матрица каждого пути;
- полный список сигнатур без случайной выборки;
- автоматически сформированный ранжированный backlog.

### 4.2 Затем общие сигнатуры

Один рабочий пакет закрывает одну повторяющуюся сигнатуру:

1. минимальный обезличенный fixture;
2. красный модульный тест;
3. общее исправление кодека;
4. зелёный модульный тест;
5. выборочная проверка на обеих базах;
6. полный прогон после группы независимых исправлений.

Запрещены:

- условия по UUID или имени объекта реальной УТ/БСП;
- обновление общего процента по выборочному прогону;
- скрывающие расхождения нормализаторы;
- allowlist как финальное состояние.

### 4.3 Очерёдность

1. Инвентарный gate и 499 пропущенных XML метаданных.
2. Lossless-представление и namespace форм.
3. Командные наборы, события, пути данных и значения по умолчанию форм.
4. MXL `format`/`formatIndex`/palette.
5. Остальные шаблоны и служебные `Ext`.
6. Корневые XML и `Configuration.xml`.
7. Полный прогон УТ и БСП.
8. Обратная загрузка через одноразовые клоны.

## 5. Параллельная работа

Чтобы агенты не конфликтовали в больших файлах:

| Поток | Владение |
|---|---|
| A — матрица | `src/plan.rs`, `src/cli.rs`, `src/main.rs`, отдельные тесты матрицы |
| B — автоматизация | `scripts/export-ibcmd-vs-ours.ps1`, новый двухбазовый скрипт, протокол |
| C — инвентарь метаданных | `src/mssql_dump/metadata.rs`, узкая маршрутизация `mod.rs`, тесты |
| D — формы | `form_body.rs`, `forms.rs`, `form_matrix.rs`, связанные тесты |
| E — MXL | `moxel.rs`, MXL-части `module_blob.rs`, связанные тесты |
| F — обратная загрузка | `src/mssql.rs`, live-тесты; начинается после стабильного экспорта |

Одновременно разрешено изменять только один высококонфликтный файл
(`mod.rs`, `form_body.rs`, `module_blob.rs`). Независимые потоки получают
разные файлы либо выполняются последовательно.

## 6. Критерии фаз

### P0 — воспроизводимость

- Полный отдельный запуск существует для УТ и БСП.
- Манифест фиксирует все версии, SHA и команды.
- Стороны не перезаписывают друг друга.
- Изменение базы во время запуска обнаруживается.

### P1 — наблюдаемость

- Каждому пути соответствует строка матрицы.
- Все агрегаты сходятся со `source-diff`.
- Каждый изменённый XML имеет сигнатуру или явную ошибку разбора.
- Markdown-таблица генерируется детерминированно из JSON.

### P2 — конвергенция

- У каждой верхней сигнатуры есть владелец, fixture и проверка на двух базах.
- Полные показатели монотонны; регрессия блокирует объединение.
- Пропущенные корневые XML равны нулю.

### P3 — точный экспорт

На УТ и БСП:

```text
different = 0
left_only = 0
right_only = 0
```

Включается `ConfigDumpInfo.xml`; финальная allowlist пуста.

### P4 — полный цикл

- XML загружается в одноразовый клон без неподдержанных строк.
- Нативная повторная выгрузка клона побайтово совпадает с исходным XML.
- `mssql-audit-source-parity` не сообщает блокеров подготовки или базовых тел.

### P5 — выпускной gate

- Переносимые тесты проходят на Windows и Linux.
- Live-матрица УТ и БСП запускается явно на исследовательской машине.
- README и отчёт получают процент только из полного валидного манифеста.

## 7. Риски и меры

- **Долгий полный прогон.** Сигнатуры проверяются выборочно, общий процент —
  только полным запуском.
- **Изменение базы между сторонами.** Одноразовый клон или проверяемый
  отпечаток до/после.
- **Корпус-зависимые исправления.** Fixtures и запрет реальных UUID в коде.
- **Потеря бинарной информации при канонизации.** Сначала lossless/opaque
  сохранение, затем типизация.
- **Конфликты агентов.** Владение файлами и один поток на крупный модуль.
- **Ложный успех нормализации.** Единственный выпускной критерий — raw SHA-256.

## 8. Отклонённые варианты

- Продолжать вести сравнение через перезапись одного Git-каталога.
- Считать выборочные BSP-прогоны доказательством для УТ.
- Исправлять по одному объекту или UUID.
- Считать `compatibility/matrix.json` таблицей реального файлового паритета.
- Исключить `ConfigDumpInfo.xml` из окончательного результата.
- Объявить готовность по семантическому, а не побайтовому совпадению.
