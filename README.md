# ibcmd-rs

> **ВНИМАНИЕ:** проект предназначен исключительно для образовательных целей.
> Его нельзя использовать в промышленной эксплуатации.
> Прямые операции с базой данных могут повредить или уничтожить данные —
> используйте их только на одноразовых тестовых копиях.

**Текущий статус:** дорожная карта автономного конвертера выполнена на
**100% (56 из 56 задач)**, но это не означает готовность к промышленной
эксплуатации. Полное сравнение УТ на `master` (`4750c20`) совпадает с
сохранённым нативным экспортом побайтово на **92,3785% (45 841 из 49 623
файлов)**; до полного совпадения остаётся **3 782 файла**. Диагностическое
полное сравнение БСП даёт **86,4568% (10 546 из 12 198 файлов)** и
**1 652** оставшихся файла. Это промежуточные snapshot-gate результаты от
24 июля 2026 года, а не заявление о выпускной готовности.

Исследовательский инструмент на Rust для создания альтернативного способа
загрузки конфигураций 1С из XML-исходников вместо медленного штатного процесса.

Первый этап был ограничен режимом чтения целевой информационной базы:

- поиск инструментов командной строки 1С;
- сканирование деревьев исходников в детерминированные манифесты;
- сравнение манифестов и построение плана загрузки;
- профилирование внешних запусков `ibcmd` или `1cv8`;
- создание шаблонов трассировки SQL Server и технологического журнала 1С;
- группировка экспортированного XML из SQL Server Extended Events по
  нормализованному тексту SQL.

Для исследованных строк хранилища теперь доступны защищённые средства прямой
записи в базу данных. Они по-прежнему предназначены только для одноразовых
тестовых баз.

## Направление автономной конвертации

Конвертер XML/CF реализован как автономное решение на Rust и при работе не
зависит от `ibcmd`, `1cv8`/Конфигуратора, EDT или JVM. Независимые оси версий,
уровни возможностей, правила обработки непрозрачных данных и политика потерь
с безопасным отказом описаны в
[контракте автономной конвертации](docs/architecture/standalone-core.md).

CF-001 завершена, цепочка парсеров CF разблокирована. Добавленный в репозиторий
корпус Format15/Format16 состоит из подготовленных вручную данных, созданных
методом «чистой комнаты»,
хранится в проверяемом виде Base64 и содержит точные координаты, SHA-256
декодированных данных, согласованный граф `root`/`version`/`versions` и не
содержит прикладного кода. В Format16 используется семантический конверт
Format15 с точным смещением `0x1359`, а не скопированная непрозрачная байтовая
константа. Дополнительный автономный контур внешнего корпуса без сетевого
доступа фиксирует независимо опубликованные файлы Format15 и Format16 по
коммиту репозитория, пути, размеру, хешу и лицензии, не включая их прикладное
содержимое в репозиторий. Подробнее:
[автономный корпус тестовых данных CF](tests/fixtures/cf/README.md).

CF-002 переносит структурное чтение Format15 в переносимый крейт `ibcmd-v8`.
Читатель сохраняет исходные заголовки файла, блоков и TOC, первоначальный
порядок записей, наблюдаемые слова заголовка `0/1/2/5`, необработанные полезные
данные и маркер отсутствующих данных как значение, отличное от пустой полезной
нагрузки. Устаревший API `module-blob` делегирует работу этому читателю и
сохраняет прежнее поведение парсера и сборщика. CF-003 добавляет транзакционную
сборку цепочек страниц: сохраняется каждый заголовок страницы, заполнение
последней страницы не включается в логические данные, а циклы, повторно
использованные адреса, частичные пересечения, выход за границы, преждевременное
окончание и неожиданное продолжение приводят к точным типизированным ошибкам.
Интеграционные тесты для двух и трёх страниц полностью выполняются в памяти.
Подробнее: [проверка читателя Format15](docs/evidence/cf-format15-reader.md) и
[проверка цепочек страниц](docs/evidence/cf-chained-pages.md).

CF-004 добавляет единое распознавание Format15/Format16 и нативный 64-битный
читатель. Полные артефакты Format16 распознаются по независимому маркеру в
стандартной позиции `0x1359` после конверта Format15. Читатель проверяет и
сохраняет эту семантическую преамбулу, а также 20-байтовые заголовки файлов,
55-байтовые заголовки блоков, 24-байтовые записи адресов, относительные
64-битные адреса страниц, порядок записей, отсутствующие данные и
многостраничные полезные нагрузки. Усечённые заголовки, некорректные маркеры,
адреса вне диапазона и арифметическое переполнение `u64` приводят к безопасному
отказу. Подробнее:
[проверка читателя Format16](docs/evidence/cf-format16-reader.md).

CF-006 добавляет ограниченное индексирование `Read + Seek` и обход вложенности
с помощью посетителя. При открытии потоковый читатель загружает только байты
TOC и имён, а также заголовки страниц; страницы данных проверяются на
допустимость диапазона и индексируются, но читаются только при выборе. На
виртуальном источнике Format15 с разреженной страницей размером 1 ГиБ доказано,
что для открытия и чтения её четырёх логических байтов выполняется менее
512 байт фактического ввода-вывода. Для вложенных хранимых контейнеров и
контейнеров raw-DEFLATE действует общий бюджет глубины, количества записей,
закодированных и декодированных байтов и коэффициента распаковки; рекурсивные
«бомбы» возвращают точное типизированное ограничение без сохранения дерева в
памяти. Подробнее:
[проверка потокового обхода](docs/evidence/cf-streaming-tree.md).

CF-007 добавляет детерминированный потоковый модуль записи Format15, которому
теперь делегирует работу устаревший сборщик контейнеров модулей. В TOC и
заголовках элементов используются компактные цепочки страниц; существующие
данные записываются на настроенные страницы фиксированного размера с
детерминированным заполнением последней страницы; отсутствующие данные
сохраняются как маркер, отличный от явно пустого блока. Распределение
планируется и проверяется по диапазонам до записи первого выходного байта,
исходные логические заголовки элементов сохраняются, а полезные нагрузки
размерами `0/1/511/512/513/4097` проходят полный цикл записи и чтения.
Повторная запись канонического результата побайтово идентична. Подробнее:
[проверка модуля записи Format15](docs/evidence/cf-format15-writer.md).

CF-008 добавляет полный модуль записи архивов Format16. Основной модуль записи
планирует 64-битные относительные смещения и цепочки страниц без материализации
таблицы страниц, передаёт объявленные полезные нагрузки из произвольных
источников `Read` через буфер 8 КиБ и сохраняет различие между отсутствующими
и пустыми данными вместе с исходными логическими заголовками. Слой CF
представляет фиксированную преамбулу `0x1359` как упорядоченные семантические
записи Format15: он может либо проверить и точно сохранить существующие байты,
либо создать новый конверт, расширив только его последнюю физическую страницу.
Тестовый файл Format16 из корпуса, созданного методом «чистой комнаты»,
в режиме сохранения
перезаписывается побайтово идентично, а при семантической регенерации
разбирается в те же упорядоченные записи и полезные нагрузки. Подробнее:
[проверка модуля записи Format16](docs/evidence/cf-format16-writer.md).

CF-009 отображает обе ревизии контейнера в нейтральный упорядоченный
`StorageImage`. Каждая запись сохраняет точное нативное имя или ключ, включая
суффиксы с точкой, исходный логический заголовок и запись адреса, упакованную и
распакованную полезную нагрузку, явно указанный вид сжатия, профиль источника и
происхождение. Суффиксы с точкой никогда не интерпретируются эвристически как
части составной записи: одна запись TOC в CF соответствует одной физической
части хранилища, а повторяющиеся нативные имена отклоняются до декодирования
полезной нагрузки. Типизированный непрозрачный атрибут CF также хранит состояние
маркера отсутствующих данных отдельно от явно пустой полезной нагрузки.
Обёртка архива сохраняет ревизию, базовое смещение, преамбулу, заголовок файла,
слова страниц и хранилища и предоставляет детерминированный поиск по `root`,
`version`, `versions` и ключам объектов. Подробнее:
[проверка отображения архива CF](docs/evidence/cf-storage-image.md).

CF-010 предоставляет доступ к этому читателю через автономные команды
`cf inspect` и
`cf verify`. Обе команды выводят версионированный JSON с распознанной
структурой, явным контрактом профиля хранилища и сжатия, перечнем выбранных
элементов, размерами и хешами SHA-256. `verify` поддерживает повторяющиеся
точные селекторы `--element`, флаг `--list-only` и повторяющиеся проверки
`--expect-sha256 NAME=HASH`. Декодируются только выбранные полезные нагрузки в
рамках общего бюджета ресурсов; способ сжатия никогда не определяется
эвристически. Сообщения об ошибках соответствуют стабильной диагностической
схеме JSON в
`stderr` и ненулевой код завершения, в том числе при пустом `PATH`:

```powershell
cargo run -- cf inspect configuration.cf
cargo run -- cf verify configuration.cf --element root --expect-sha256 root=<sha256>
cargo run -- cf verify configuration.cf --element versions --list-only
```

Подробнее: [проверка автономного CLI для CF](docs/evidence/cf-cli.md).

CF-011 добавляет детерминированную переупаковку без потерь для обеих ревизий
контейнера. Перед записью проверяется, что каждая запись CF состоит ровно из
одной физической части, имеет точное имя из исходного заголовка,
типизированное состояние отсутствия или наличия, поддерживаемый способ сжатия
и согласованные упакованные и распакованные данные. Переупаковка сохраняет
порядок источника,
исходные логические заголовки, слова структуры Format15/16, точную преамбулу
Format16 и неизменные упакованные байты. Для атомарного создания нового файла
используются временный файл в том же каталоге, сброс и синхронизация данных,
повторное открытие и проверка каждой записи, а затем устойчивая к гонкам
публикация без перезаписи. При ошибке промежуточный результат удаляется, а
существующий файл назначения намеренно не заменяется. Подробнее:
[проверка переупаковки CF без потерь](docs/evidence/cf-lossless-repack.md).

CF-012 подключает декодированный `StorageImage` к существующим декодерам
семейств исходников без зависимостей от SQL, 1С, EDT, Java или дочерних
процессов. Физические записи из нескольких частей группируются в стабильном
порядке источника и передаются через единую границу, не зависящую от
поставщика; устаревший путь получения данных из MSSQL остаётся отдельным и
сохраняет поведение с немедленным отказом. `cf export` выводит иерархический XML
для известных семейств и версионированный JSON-отчёт по каждой записи со
статусами `supported`, `opaque` и `failed`. Неизвестные записи сохраняются как
явные непрозрачные результаты, а не угадываются и не отбрасываются без
уведомления. Добавленный в репозиторий CF из корпуса, созданного методом
«чистой комнаты», и полная
синтетическая запись Configuration экспортируются даже при пустом `PATH`:

```powershell
cargo run -- cf export configuration.cf source --source-version 2.20
```

Подробнее:
[проверка автономного экспорта CF](docs/evidence/cf-offline-export.md).

CF-014 добавляет наложение XML/исходников того же профиля на базовый CF.
`ibcmd-cf` выполняет полную предварительную проверку целевого хранилища,
заменяет скомпилированные записи на месте, разрешает `NeedsBase` только через
явно переданный вызывающей стороной кодек, отклоняет каждый результат
`Unsupported` и сохраняет порядок, заголовки, атрибуты и точные упакованные
байты неизвестных или неизменённых записей. Корневой адаптер предоставляет
существующие кодеки метаданных на Rust и исправляет нативную карту `versions`
для каждого изменённого ключа. При публикации повторно используется путь
из CF-011: временная запись без перезаписи в том же каталоге, повторное
открытие и полная проверка. Модули, необработанные ресурсы и зависящие от базы
XML-файлы `CommandInterface` проходят сквозной бинарный тест при пустом `PATH`:

```powershell
cargo run -- cf overlay base.cf changed.cf `
  --module '<storage-key>=CommonModules/Demo/Ext/Module.bsl' `
  --raw-asset '<storage-key>=CommonPictures/Logo/Ext/Picture.bin' `
  --command-interface '<storage-key>=Subsystems/Main/Ext/CommandInterface.xml' `
  --source-version 2.20
```

К повторяемым селекторам исходников также относятся `--metadata-xml` и
`--common-module-xml`. Ключи хранилища остаются явными: ни имена файлов, ни
суффиксы UUID не угадываются. Подробнее:
[проверка наложения CF](docs/evidence/cf-overlay.md).

CF-015 усиливает защиту всей границы контейнера с помощью добавленного в
репозиторий корпуса некорректных данных, созданного методом «чистой комнаты»,
для заголовков,
шестнадцатеричных полей, смещений, цепочек страниц, имён UTF-16, полезных
нагрузок DEFLATE и вложенных архивов. Детерминированные тесты свойств создают
документы Format15 и Format16 на границах страниц, сохраняют различие между
отсутствующей и пустой полезной нагрузкой и обходят вложенные деревья
хранилища. Отдельная цель `cargo-fuzz` пропускает ограниченные произвольные
байты через единое индексирование, выборочное чтение, декодирование полезной
нагрузки и обход вложенности; это подключаемое рабочее пространство
разработчика, а не промышленная зависимость. Обычный контур CI для
Linux/Windows всегда запускает детерминированный набор тестов повреждений и
свойств. Подробнее:
[проверка защиты контейнера CF](docs/evidence/cf-container-hardening.md).

CF-005 завершена независимо от недоступных эталонных контейнеров. Слой полезной
нагрузки CF теперь отличает `Stored` от одного полного потока raw-DEFLATE,
требует `StreamEnd` и полного потребления закодированных входных данных и через
типизированные ошибки отклоняет усечённые потоки, лишние завершающие байты,
превышение суммарных ограничений глубины, количества или объёма и чрезмерный
коэффициент распаковки. При декодировании используются буферы кодека
фиксированного размера, а общий бюджет ресурсов учитывается только после полной
проверки; регрессионные тесты синтетических повреждений и «бомб» распаковки
выполняются без 1С, EDT и JVM. Подробнее:
[проверка ограниченной полезной нагрузки CF](docs/evidence/cf-payload-limits.md).

BOOT-002 завершена. Ограниченный граф идентификаторов и состава, а также
детерминированные `root`, `version`, `versions` и семисекционное тело
`Configuration`, ограниченное профилем, компилируются без базового артефакта.
Модуль записи Configuration выводит семь разделённых по доменам идентификаторов
UUIDv8, направляет 47 известных семейств верхнего уровня в 48 точных нативных
позиций, проверяет типизированные межобъектные ссылки и до начала записи
отклоняет неизвестное семейство или будущую структуру. Реализация и тесты не
запускают 1С, EDT, JVM, SQL Server или какие-либо внешние процессы. Подробнее:
[проверка тела Configuration](docs/evidence/configuration-body-8.3.27.md).

BOOT-003 завершена. Шесть её вертикальных срезов — `Constant`, `Language`,
`SessionParameter`, `DefinedType`, `FunctionalOption` и
`FunctionalOptionsParameter` — поддерживают цепочку XML 2.20/2.21 ->
типизированное IR -> детерминированная нативная строка без базового артефакта
-> строгое нативное IR/XML для независимо выбранных структур семейств 8.3.27.
Читаемые ссылки `Location`, `Content` и `Use` разрешаются только через
проверенный канонический граф; значения шаблонов типов `cfg:*` разрешаются
только через канонические сгенерированные TypeId. Ядро теперь также сохраняет
сгенерированный `ValueId` как типизированный необязательный идентификатор
(устаревший канонический JSON остаётся читаемым; домен семантического дайджеста
— v2). Кодек `Constant` сохраняет все три сгенерированные пары
TypeId/ValueId, единственный шаблон типа и `UseStandardCommands`; неизвестные
поля нативной оболочки и не подтверждённые данными сопоставления квалификаторов
приводят к безопасному отказу.

Все подтверждённые текстовые нативные строки BOOT-003/BOOT-004 содержат
обязательный UTF-8 BOM, за которым следует детерминированный текст с CRLF;
сериализаторы и строгие декодеры используют единый байтовый контракт уровня
профиля.

BOOT-004 завершена. Все семь срезов семейств сервисов — `ScheduledJob`,
`EventSubscription`, `HTTPService`, `WebService`, `IntegrationService`,
`WSReference` и `XDTOPackage` — теперь поддерживают цепочку XML 2.20/2.21 ->

APP-001 добавляет сервис и CLI верхнего уровня `convert`. Формат артефакта и
точный профиль версии являются отдельными обязательными координатами; сборка
платформы не выводится из версии XML. Каждый поддерживаемый маршрут сообщает об
одинаковых этапах: `decode -> validate -> migration_plan -> migrate ->
encode_preflight -> atomic_encode`. Флаг `--dry-run` завершает предварительную
проверку без публикации результата, `--loss error|warn|drop` передаётся в
защищённое ядро миграции, а `--report` записывает тот же стабильный JSON, который
выводит команда. Новые результаты публикуются атомарно без перезаписи:

```powershell
cargo run -- convert source configuration.cf `
  --source-format xml --target-format cf `
  --source-profile xml-2.20 --target-profile platform-8.3.27.1989

cargo run -- convert configuration.cf restored `
  --source-format cf --target-format xml `
  --source-profile platform-8.3.27.1989 --target-profile xml-2.20

cargo run -- convert source migrated `
  --source-format xml --target-format xml `
  --source-profile xml-2.20 --target-profile xml-2.21 `
  --dry-run --report conversion-report.json
```

Сервис также предоставляет переупаковку CF без потерь в пределах одного
профиля. Межпрофильные маршруты XML остаются ограничены проверенными
направлениями миграции; экспорт CF разрешает только точно регенерированные

запись блокирует публикацию. Подробнее:
[проверка сервиса автономной конвертации](docs/evidence/offline-conversion-service.md).

### Совместимость с клиентом bcp

Команды `mssql-storage-*` и `mssql-delta-*` запускают `bcp` как внешний процесс.
Нативная команда `mssql-dump-config` также использует существующий нативный
парсер `bcp` для получения содержащих двоичные объекты строк при потоковом
экспорте, тогда как облегчённые запросы заголовков и управляющие запросы
по-прежнему выполняются через `sqlcmd`. По умолчанию в этих сценариях `bcp`
запускается в нативном формате, который поддерживается разными версиями `bcp`,
включая `bcp 13` (Microsoft ODBC Driver 13 for SQL Server).

`bcp 18+` (ODBC Driver 18) по умолчанию использует зашифрованные соединения,
поэтому может потребоваться доверие к сертификату сервера. В таком случае
передайте `--bcp-trust-cert`, чтобы добавить `bcp -u` (доверять сертификату
сервера). **Не** используйте этот флаг с `bcp 13` и более ранними версиями:
такие сборки не распознают `-u` и завершаются ошибкой синтаксиса вызова
с сообщением `"unknown argument"`.

```powershell
# bcp 18+ через зашифрованное соединение с сервером с самоподписанным сертификатом:
cargo run -- mssql-storage-export --database import_only_db -o storage-bundle --overwrite --bcp-trust-cert
```

## Первый эксперимент с ERP

1. Подготовьте временную информационную базу ERP на SQL Server.
2. Экспортируйте или подготовьте XML-исходники ERP.
3. Создайте базовый манифест:

   ```powershell
   cargo run -- scan C:\erp-src -o baseline-manifest.json
   ```

4. Сгенерируйте шаблоны трассировки:

   ```powershell
   cargo run -- trace-template .\trace
   ```

5. Запустите SQL Server Extended Events из `trace\sqlserver-xevents.sql`.
6. Выполните медленную загрузку через `profile-run`:

   ```powershell
   cargo run --features platform-oracle -- profile-run --capture-output -- ibcmd ...
   ```

7. Остановите трассировку SQL и сохраните вместе `.xel`, технологический журнал
   1С, манифест и JSON-файл профиля.
8. Экспортируйте XML событий из SQL Server и сгруппируйте его:

   ```powershell
   cargo run -- trace-analyze C:\temp\events.xml -o trace-analysis.json
   ```
9. Сопоставьте сгруппированный SQL с семействами изменений хранилища:

   ```powershell
   cargo run -- storage-map C:\temp\events.xml -o storage-map.json
   ```

## План развития

1. Модель исходников: разбор идентификаторов объектов, UUID и принадлежности
   модулей, форм и шаблонов.
2. Мост пакета хранилища: экспорт и импорт `ConfigSave` и `Params` через
   нативный клиент BCP для SQL Server, чтобы подготовленное состояние импорта
   можно было
   применить к пустой информационной базе.
3. Мост пакета дельты: экспорт и импорт подготовленных строк `ConfigSave`
   для заранее сформированного частичного изменения существующей
   информационной базы.
4. Компилятор тела общего модуля: создание корректного двоичного объекта `.0`
   модуля из исходного кода BSL в виде `deflate(V8File(info,text))`
   с использованием базового двоичного объекта для заголовков элементов.
5. Модификатор `versions`: создание подготовленного двоичного объекта
   `versions` путём замены поколения, необязательных записей
   `root`/`version`/`versions` при их наличии и UUID изменённых файлов;
   подготовка исходников и метаданных может добавлять отсутствующие записи
   об изменениях для новых подготовленных строк тел.
6. Подготовщик общих модулей в SQL: чтение активного `Config`, создание
   изменённых двоичных объектов `.0` и `versions` и запись набора из пяти
   подготовленных строк `ConfigSave`.
7. Подготовщик нескольких модулей: подготовка нескольких изменений тел общих
   модулей в одном наборе `ConfigSave` с единым модифицированным двоичным
   объектом `versions`.
8. Подготовщик метаданных общего модуля: подготовка XML-изменений `Name`,
   `Synonym`, `Comment`, флагов контекста выполнения, `Privileged`
   и `ReturnValuesReuse` с помощью набора из четырёх строк `ConfigSave`.
9. Подготовщик объекта общего модуля: подготовка целого общего модуля из XML
   и соседнего `Ext\Module.bsl` в одном наборе из пяти строк `ConfigSave`.
10. Пакетный подготовщик объектов общих модулей: подготовка нескольких целых
    общих модулей из XML и соседних файлов `Ext\Module.bsl` с одним общим
    двоичным объектом `versions`.
11. Универсальный подготовщик простых метаданных: подготовка XML-изменений
    только для метаданных `Name`, `Synonym` и `Comment` с сохранением остальной
    части каждого двоичного объекта метаданных; проверено на `Constant`
    и `SessionParameter`. Для `Constant` также подготавливаются поддерживаемые
    шаблоны `Type` (`boolean`, `string`, `decimal`, `dateTime`)
    и `UseStandardCommands`. Для `DefinedType` подготавливаются встроенные
    шаблоны `Type` с одной или несколькими записями `boolean`, `string`,
    `decimal` и `dateTime`, а также ссылочные типы `cfg:*`, определяемые
    по сгенерированным значениям `TypeId` в `--source-root`. Для
    `CommonCommand` подготавливаются `Representation`, `ToolTip`,
    `IncludeHelpInContents`, `ParameterUseMode`, `ModifiesData`, `Picture`
    для пустых ссылок или ссылок `CommonPicture.<name>`,
    `CommandParameterType` для пустого значения или одного
    `cfg:DefinedType.<name>`, а также наблюдаемое в настоящее время значение
    `Auto` свойства `OnMainServerUnavalableBehavior`. Для разрешения ссылок
    требуется `--source-root`; `StdPicture.User` сопоставляется с принадлежащим
    платформе UUID изображения пользователя, а остальные значения
    `StdPicture.*` и произвольные многотиповые наборы параметров команд
    по-прежнему отклоняются до появления соответствующих сопоставлений.
12. Проверка SQL: сравнение структуры таблиц, количества строк, а в дальнейшем
    и контрольных сумм строк.
13. Анализатор трассировки: расширение поддержки экспорта `.xel`
    и добавление более надёжной нормализации SQL.
14. Сопоставитель хранилища: сопоставление операций с метаданными 1С
    с наблюдаемыми изменениями SQL для каждой версии платформы.
15. Усиление защиты записи: сохранение разрушающих операций подготовки
    за явными флагами подтверждения и добавление дополнительных защитных
    барьеров перед использованием вне лабораторной среды.
16. Матрица совместимости: маршруты автономной конвертации теперь определяются
    на основе точных результатов тестов профилей и семейств; исследование
    работающей СУБД остаётся отдельной задачей `platform-oracle`.
