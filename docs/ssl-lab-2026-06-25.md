The module body is a raw deflate stream. After inflation, it is a V8 container
with two plain elements:

- `info` - UTF-8 BOM text `{3,1,0,"",0}`.
- `text` - the BSL file bytes, including UTF-8 BOM and CRLF.

Important details:

- The outer SQL blob is raw deflate over the whole inner V8 container.
- The inner V8 container has `storage_ver = 1`.
- Inner `info` and `text` element data must be stored plain, not deflated again.
- The `info` data block uses page size 512; `text` used page size equal to the
  text length in this module.
- `v8unpack --build` was not directly suitable because it deflated inner
  elements; `config check` tolerated that, but `config apply` failed with
  `Ошибка формата потока`.

## Current Interpretation

The XML import phase writes compiled internal configuration blobs into `ConfigSave`.
After `apply`, the active configuration is moved into `Config` and `ConfigSave`
is emptied. The raw XML files are not stored directly as-is. Therefore the next
replacement milestone is not plain XML file insertion; it is either:

1. reproduce the XML-to-internal-binary compiler for a small metadata subset, or
2. isolate a smaller operation, such as one common module change, and implement
   the corresponding SQL mutation behind an explicit unsafe test-only command.

The `mssql-storage-export/import` commands are a SQL-level bridge: they replace
`ibcmd config import` once an import-state bundle exists, but they do not yet
compile XML sources into that bundle.

The `mssql-delta-export/import` commands move staged `ConfigSave` rows into
another compatible infobase. For this tested common module body change,
`mssql-stage-common-module` and `mssql-stage-common-modules` now create the
staging set directly in SQL Server without `ibcmd config import files`.
`mssql-stage-common-module-metadata` does the same for common-module XML
metadata: `Name`, `Synonym`, `Comment`, execution-context flags, `Privileged`,
and `ReturnValuesReuse`. `mssql-stage-common-module-object` now stages one
complete common module object from XML plus sibling `Ext\Module.bsl`.
`mssql-stage-common-module-objects` stages several such objects in one apply
cycle. `mssql-stage-metadata-objects` now stages metadata-only XML changes for
simple objects by patching only the common `Name`, `Synonym`, and `Comment`
fields while preserving the rest of each active metadata blob. For constants,
it also patches the value type pattern and `UseStandardCommands`. For defined
types, it patches builtin type patterns and `cfg:*` references resolved through
`--source-root`. For common commands, it patches the command presentation,
tooltip, help, parameter-use and data-modification flags, plus
empty/CommonPicture picture references and empty/single-DefinedType command
parameter types. `Group` is still preserved from the active blob.

The next technical target is widening that metadata compiler from header fields
to type descriptions, command properties, child objects, forms, templates, and
eventually a full XML source tree.
